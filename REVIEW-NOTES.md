## Deliberate deferrals — disagree-and-justify

### [correctness #2] CRLF line endings break `/^---$/` matching

**Critique (correctness persona):** *"A SKILL.md saved with CRLF will get 'no frontmatter found' — misleading error. Strip `\r` in awk preprocessing or document CRLF as out-of-scope."*

**Decision:** Ship despite — documented as out-of-scope in spec rather than handled in code.

**Reasoning:** Git's default LF normalization with `text=auto` attributes handles this for the vast majority of contributors. The persona linked this concern to commit `2cc232a` (Windows-side setup), but that commit is about toolchain (`migrate.sh` + symlink helpers for Windows users), not about Windows authors editing SKILL.md files. SKILL.md files flow through git, which normalizes them. Adding awk preprocessing for an edge case Git already handles introduces code that masks misconfiguration (a contributor with `core.autocrlf` wrong should know). Mitigation: explicit out-of-scope note added to `skills/_format/FRONTMATTER-SPEC.md § Out of scope for v1` pointing to `git config core.autocrlf` and `.gitattributes` remedies.

Issue #04 (schema validator slice) extends the YAML grammar handling — that's the natural slice to land these fixtures alongside any new grammar additions, since both would be tested as one cohesive unit.

### [security #1] SHA-pin `actions/checkout@v4`

**Critique (security persona, original):** *"`actions/checkout@v4` is pinned to a mutable major tag rather than a full commit SHA — OWASP/SLSA-recommended supply-chain posture."*

**Decision:** Recalibrated during persona dialogue (score moved 9.4 → 9.6). Future-tightening candidate, not a PR action item.

**Reasoning (from dialogue):** The workflow has no secrets, uses `pull_request` (not `pull_request_target`), and runs in the read-only sandbox. SHA-pin discipline carries maintenance cost (manual SHA bumps; risk of drifting onto known-vulnerable SHAs if Dependabot PRs get missed). For *this* workflow, cost > benefit. The right time to adopt SHA-pinning is when a secret-bearing or `pull_request_target` workflow lands; that's a repo-wide policy decision deserving its own review, not a drive-by on the first PR. The persona explicitly recalibrated and acknowledged this framing.

### [security #2] `find` follows symlinks

**Critique (security persona):** *"`find $TARGET -name SKILL.md` will follow symlinks out of the repo if $TARGET contains a symlinked directory; theoretical, but adding `-not -path '*/.*'` would prevent surprise traversal."*

**Decision:** Ship despite.

**Reasoning:** The persona's own caveat ("theoretical") and the fact that the validator runs on trusted checkouts (CI clones a known commit) make this a hypothetical risk. The cost of adding the flag is small, but the risk model doesn't apply — for an actual attack to occur, an adversary would need write access to a repo's source, at which point they have many easier vectors. Defer to a future hardening pass if/when validate.sh is invoked in less-controlled contexts.

### [backwards-compat #3] Consider `--legacy` / `--warn-only` mode

**Critique (backwards-compat persona):** *"A warn-only mode would give adopters a one-release grace period to clean up custom fields."*

**Decision:** Ship despite — strict mode only in v1.

**Reasoning:** Hits the rubric's "Suggesting unrelated refactors that expand scope" anti-pattern. Strictness is the point of the v1 design (catches typos like `requires-skils`). Adding a `--warn-only` mode would dilute the contract and create two enforcement levels to maintain. The migration documentation (now in `skills/_format/FRONTMATTER-SPEC.md`) is the right tool for the v1 transition. If adopter pain surfaces during the MMC pilot, that's signal to revisit; for now, ship strict.
